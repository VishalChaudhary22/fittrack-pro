Product decisions

→Keep search fully local for speed and zero API cost.
→Separate base foods from recipes/dishes for cleaner nutrition logic.
→Store computed macros on each log entry so history remains stable even if the database changes later.
→Allow custom entries for homemade or restaurant food from day one.

Migration rules

→If an old entry has only calories, keep it renderable as a legacy item.
→Use safe fallbacks: protein/carbs/fat/fiber default to 0 for old entries.
→Mark legacy entries with sourceType: 'legacy' so the UI can label them.
→Never recalculate old logged macros from updated food data unless the user explicitly edits the item.
Phase 1 — Data foundation
Build a trustworthy v1 dataset and utilities before any UI polish.
  • Create 100–150 high-frequency foods first across dal, roti, rice, veg, dairy, eggs/chicken/fish, snacks, fruits, drinks, and supplements.
  • Define category constants, serving conventions, and validation helpers.
  • Add a dataset audit script for duplicate ids, missing grams, invalid macros, and empty tags.
Phase 2 — FoodLogPage MVP
Ship the core behavior end to end.
  • Search by name and alt names.
  • Select meal slot, serving, quantity, or custom grams.
  • Log entry, edit entry, delete entry, and group by meal.
  • Show daily totals for calories, protein, carbs, fat, and fiber.
Phase 3 — Integration and retention
Make it useful every day inside FitTrack Pro.
  • Dashboard mini-bars for Cals, P, C, F.
  • Diet page "Log Food" CTA and remaining macro hints.
  • Recent foods, favorites, quick-add from yesterday, and date navigation.
Phase 4 — Smarter nutrition layer
Only after MVP is stable.
  • Weekly macro summary and adherence chart.
  • Recipe builder for homemade dishes.
  • Goal-aware suggestions like protein-first or low-fat options.
  • Optional photo logging later, not in MVP.

Gemini 3.1 Pro High Large-file reasoning, feature implementation with multiple files, code review with context, drafting exact change plans. Tiny UI text changes or repetitive dataset expansion. Use when task spans several files but does not need Opus-level deep reasoning.
Gemini 3.1 Pro Low Single-feature coding, medium-complexity CRUD work, generating helper utilities, route wiring, test scaffolds. Messy bug hunts across the whole app. Make this your default coding model.
Gemini 3.1 Flash Summaries, prompt cleanup, changelogs, TODO extraction, file inventory, converting notes into concise tasks. Core logic, migrations, or important implementation. Use for prep and compression, not final coding.

Optimal workflow

1Use Flash to condense your notes into a clean task brief before you start a coding session.
2Use Pro Low as the default builder for one well-scoped unit of work.
3Escalate to Pro High only when the task spans multiple files or the first pass is weak.
4Bring in Opus 4.6 for architecture, migrations, or final review of risky code.
5After each completed feature, ask Flash to summarize what changed and update your running spec.

Context rules to reduce token burn

→Do not paste the entire project every time; paste only the relevant files plus a short project brief.
→Keep one master build note with architecture, decisions, and "do not change" constraints.
→Ask for plan first, approve it, then request code.
→Batch only related changes together; never mix unrelated UI, data, and auth tasks in one prompt.
→After merge, start a new thread for the next major task using the updated brief.
